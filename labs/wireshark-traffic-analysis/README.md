# Wireshark Traffic Analysis Lab

## Objective
To capture and analyze live network traffic using Wireshark, identify 
common protocols, inspect packet contents, and understand how data 
moves across a network at the packet level.

## Tools Used
- Wireshark (Windows)
- Wi-Fi network interface

---

## What I Did

### 1. Live Traffic Capture
Started a live capture on my Wi-Fi interface and observed real-time 
traffic generated by my machine. Captured 764 packets over a short 
session, identifying a mix of TLSv1.2, TCP, and UDP protocols. Observed 
active connection terminations via FIN/ACK packets between my machine 
(192.168.100.8) and multiple external IPs.

### 2. HTTP Filter — Why So Little HTTP?
Applied an `http` display filter expecting to see browser traffic. Only 
2 packets appeared — both from Chrome fetching a certificate revocation 
file from `cert.ssl.com` over plain HTTP. This highlighted a key 
real-world finding: the vast majority of modern web traffic is encrypted 
over HTTPS/TLS and is invisible to HTTP filters, making unencrypted HTTP 
increasingly rare on modern networks.

<img width="1920" height="836" alt="Http Filter" src="https://github.com/user-attachments/assets/e4a62662-bb51-4a32-9197-e0a395158b0a" />


### 3. Packet Inspection — Plaintext HTTP Headers
Inspected one of the two HTTP packets in detail. The packet was a GET 
request to `cert.ssl.com` on port 80 (destination IP: 3.5.22.194). 
Expanding the HTTP layer revealed the full request headers in plaintext 
including the Host, User-Agent (Chrome 145.0 on Windows 10 x64), 
Accept-Encoding, and Accept-Language fields. This demonstrated how 
unencrypted HTTP exposes detailed client information to anyone 
monitoring the network.

### 4. DNS Query Analysis
Applied a `dns` filter and intentionally opened Roblox to generate 
DNS traffic. Captured a DNS query from my machine (192.168.100.8) to 
my router/DNS resolver (192.168.100.1) on port 53, requesting the A 
record for `ncs.roblox.com`. The query packet was only 74 bytes, 
containing no answer records — purely the outbound request.

<img width="1524" height="752" alt="Dns Query 2" src="https://github.com/user-attachments/assets/60025419-c6ba-49dc-a412-fdcb78b900e0" />


### 5. DNS Response Analysis
Captured the corresponding DNS response from the resolver back to my 
machine. The response revealed a CNAME chain: `ncs.roblox.com` resolved 
to `edge-term4.roblox.com`, which further resolved to 
`edge-term4-mia4.roblox.com` — Roblox's CDN edge server located in 
Miami. The response also included an authoritative nameserver record 
pointing to `dns1.p06.nsone.net`. Total response time was 10.76ms. 
This demonstrated how DNS can expose background application activity 
even when no browser is open.

<img width="1525" height="749" alt="DNS Response" src="https://github.com/user-attachments/assets/b4ff11bb-7e9c-4863-aab5-e4711847e5ac" />


### 6. Protocol Hierarchy
Used Statistics → Protocol Hierarchy to analyze the full breakdown of 
a capture session containing 11,024 packets. Key findings:
- **QUIC (78.1% of packets, 89.4% of bytes)** — Google's modern 
  transport protocol dominated traffic, used by Chrome, YouTube, 
  and Google services
- **TLS (5.8%)** — encrypted HTTPS traffic
- **DNS (0.8%)** — domain resolution requests
- **HTTP (0.0%)** — only 4 packets, confirming how rare unencrypted 
  traffic is on a modern network
- IPv4 accounted for 98.8% of traffic vs IPv6 at 1.1%

## Key Takeaways
- Modern networks are dominated by encrypted traffic (QUIC + TLS), 
  making traditional HTTP packet inspection largely obsolete for 
  monitoring web browsing
- Unencrypted HTTP still exposes full request headers in plaintext 
  including browser version, OS, and accepted content types — a 
  significant information leak on unsecured networks
- DNS queries reveal background application activity even when no 
  browser is open — Roblox was generating DNS lookups passively
- CNAME chains in DNS responses show how large platforms use CDN 
  infrastructure to route users to geographically close servers
- Protocol Hierarchy statistics provide a fast high-level fingerprint 
  of network behaviour without reading individual packets
